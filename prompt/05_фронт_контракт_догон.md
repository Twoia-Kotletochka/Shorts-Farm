## Что изменилось (2 фичи + мелочи)

### 1) Балансир LLM — списки моделей (Настройки → AI-провайдеры → LLM)
`llm_provider` теперь, помимо `type / base_url / api_key / model / model_fast`, имеет **два упорядоченных списка**:
```ts
llm_provider: {
  type: string; base_url?: string; api_key?: string;
  model: string;            // основная сильная модель (back-compat)
  model_fast?: string;      // основная быстрая (1-й проход)
  models?: string[];        // НОВОЕ: сильные модели по приоритету (2-й проход/метаданные)
  models_fast?: string[];   // НОВОЕ: быстрые модели по приоритету (1-й проход)
}
```
Семантика: бэкенд пробует модели по порядку и при лимите/кредитах/подписке/ошибке (402/403/429) **переключается на следующую** (в т.ч. бесплатную). Пусто → используется одиночный `model`/`model_fast`.

**UI:** в карточке LLM-провайдера — редактируемый **упорядоченный список** моделей:
- «Модели (по приоритету)» → `models`; «Быстрые модели (по приоритету)» → `models_fast`.
- Добавить / удалить / перетащить порядок (чипы или список со стрелками). Первый элемент — основной.
- На сохранение слать массивы `models`/`models_fast`. STT-провайдер не трогаем (одна модель).
- Пресеты можно дополнить готовыми списками (напр. для Ollama Cloud: сильные `["gpt-oss:120b","gemma3:27b","gpt-oss:20b"]`, быстрые `["gpt-oss:20b","gemma3:12b","ministral-3:8b"]`).

### 2) Выбор аудиодорожки (Создание шортсов; индикатор в Библиотеке)
`GET /api/movies` и `GET /api/movies/{id}` теперь возвращают `audio_tracks`:
```ts
type AudioTrack = {
  index: number;          // относительный индекс среди аудио — ИМЕННО его слать в задачу
  stream_index?: number;
  language: string | null; // ISO-639-2, напр. "rus" / "eng"
  title: string | null;
  channels: number | null; // 2, 6 (5.1) ...
  codec: string | null;    // aac, ac3, dts ...
  default: boolean;
};
// MovieOut.audio_tracks: AudioTrack[]
```
`POST /api/jobs` и `POST /api/jobs/batch` принимают новое необязательное поле:
```ts
audio_track?: number | string | null;  // index дорожки ИЛИ код языка ("rus"); null/опустить → авто
```
Бэкенд при «авто» сам выбирает: язык контента → дорожка с флагом default → макс. каналов.

**UI (визард, шаг после выбора фильма):**
- Селект **«Звуковая дорожка»**, наполняется из `movie.audio_tracks`.
  - Первый пункт — **«Авто (по языку)»** → в теле задачи `audio_track` опустить/`null`.
  - Остальные — по дорожке, подпись вида: `#0 · RUS · 5.1 · ac3 · по умолчанию` (из `index/language/title/channels/codec/default`).
  - Выбор дорожки → слать `audio_track: <index>` (число).
- Если `audio_tracks` пуст (файл ещё не пробирован / один поток / статус ошибки) — показывать только «Авто» (или скрыть селект).
- В Библиотеке полезен бейдж «N дорожек» для много-дорожечных файлов.

### 3) Мелочи контракта (проверь, что учтено)
- `GET /api/settings` отдаёт `panel_password_set: boolean` вместо самого пароля. PUT принимает `panel_password` — слать только при изменении; пустая строка `""` отключает пароль; не слать → не менять.
- `api_key` провайдеров на GET всегда **маскирован** (`"****abcd"`). На PUT: новый ключ слать только если меняешь — маска/пусто **сохраняют** прежний ключ (та же логика теперь и при изменении `models`/`models_fast` без ключа).
- `POST /api/providers/test` без изменений (`{ kind, config }`), проверяет **одну** модель (`config.model`). Для балансира можно тестировать первую из `models` или дать кнопку на каждую — но это опционально.

## 4) Удаление задач очереди (вкладка «Очередь»)
Новые эндпоинты:
- `DELETE /api/jobs/{id}?delete_shorts=<bool>` → `{ ok: true, deleted_shorts: number }`.
  Удаляет задачу целиком (для сломанных/ненужных/«передумал»). Активную сначала отменяет.
  `delete_shorts=false` (по умолчанию) — **готовые ролики сохраняются** (отвязываются от задачи,
  остаются в «Готовых»); `delete_shorts=true` — ролики этой задачи удаляются вместе с файлами.
- `POST /api/jobs/bulk` `{ ids: number[], action: "delete" | "cancel", delete_shorts?: boolean }`
  → `{ ok: true, affected: number }` — массово.

**UI:**
- Отличать **две** операции у задачи:
  - «Отменить» (`POST /api/jobs/{id}/cancel`) — остановить выполнение, **запись задачи остаётся** (статус canceled);
  - «Удалить» (`DELETE /api/jobs/{id}`) — **убрать задачу из списка совсем**.
- Кнопка «Удалить» с подтверждением. Если у задачи есть готовые ролики (status done и шортсы есть) —
  в диалоге чекбокс **«Удалить и готовые ролики этой задачи»** → `delete_shorts=true`; иначе ролики
  останутся в «Готовых».
- Массовые действия: выбор нескольких задач + «Удалить выбранные» / быстрый «Очистить завершённые/упавшие»
  через `POST /api/jobs/bulk` (action `delete`). Можно также массовый `cancel`.
- После удаления — перезапросить `/api/jobs` (и `/api/shorts`, если удаляли ролики).

## 5) Режим ролика: single vs compilation — РАЗВЕСТИ ЯВНО (частая путаница)
В `POST /api/jobs` поле `format`:
- `format: "single"` (**должно быть значением по умолчанию**) — `count` ОТДЕЛЬНЫХ роликов, каждый
  длиной `target_duration_sec` (напр. 3 ролика по 30–45с). → N файлов.
- `format: "compilation"` — **ОДИН** файл-монтаж из нескольких коротких хайлайтов.
  Здесь `count` = сколько моментов СКЛЕИТЬ в один ролик (не число файлов!), а `target_duration_sec`
  для компиляции **не** длина клипа. Длиной управляют (бэкенд-дефолты, можно вынести в UI позже):
  `compilation_segment_sec` (длина каждого момента, деф. [6,12]) и `compilation_total_sec`
  (общая длина монтажа, деф. 60).

**UI визарда:**
- Тумблер «Тип ролика»: **«Отдельные ролики» (single, по умолчанию)** / «Компиляция».
- Подписи, снимающие путаницу:
  - single: «`count` = сколько роликов», «`target_duration_sec` = длина каждого».
  - compilation: «один ролик из нескольких моментов», «count = сколько моментов», и (опц.)
    поля «длина момента» / «длина всего ролика».
- Проверь, что при «Отдельные ролики» в тело уходит `format: "single"` (а не compilation) —
  была ситуация, когда выбирали «3 ролика», а уходило `compilation` → получался один длинный файл.

## 6) Авторизация по паролю панели (НОВОЕ — теперь реально enforced)
Если в Настройках задан `panel_password`, бэкенд требует его на ВСЕХ эндпоинтах, кроме
`/api/health` и `/api/auth/*`. Иначе — 401.
- `GET /api/auth/status` → `{ password_required: boolean }` — вызвать на старте; если true и нет
  сохранённого пароля → показать экран входа.
- `POST /api/auth/login { password }` → `{ ok: true }` или 401 — проверка пароля на входе.
- Способ передачи пароля в запросах:
  - обычные fetch/axios — заголовок **`X-Panel-Password: <пароль>`** (добавить в API-клиент по умолчанию);
  - для медиа в `<video src>`, `<img>` и ссылок скачивания (заголовок не повесить) — query-параметр
    **`?panel_key=<пароль>`** к `/shorts/{id}/preview|file|thumbnail`.
- Хранить пароль в памяти/sessionStorage после успешного `login`; при 401 — разлогинить и показать вход.

## 7) Видимость ошибки финального рендера
`GET /api/shorts/{id}` (и metadata): если финальный рендер упал, в `metadata.render_error` будет
строка с причиной. На карточке шортса со статусом `approved`, но без `has_final`, показать эту ошибку
(и дать «повторить»: `POST /api/shorts/{id}/approve` снова ставит финал-рендер).

## 8) Новый жизненный цикл шортса (ВАЖНО — много поменялось в поведении)
Бэкенд переделал флоу так, чтобы субтитры были видны ещё до одобрения, обрезка реально
применялась, а одобренный был финалом макс. качества.

- **Субтитры теперь ПРОЖИГАЮТСЯ в черновик** (preview уже WYSIWYG: 9:16 + субтитры). Мягкий
  оверлей рисовать поверх плеера больше НЕ нужно (эндпоинт `/subtitles` остаётся как справочный).
- **Обрезка краёв**: `PATCH /api/shorts/{id} { start_ts?, end_ts? }` — бэкенд ПЕРЕ-РЕНДЕРИТ превью
  с новыми границами (асинхронно, ~10–40 c; если язык субтитров ≠ языка контента, идёт перевод).
  Аналогично `subtitles_text` (ручная правка субтитров) → пере-рендер.
- **Поле `rev`** в `ShortOut` (версия файла, меняется после ре-рендера). Добавляй к URL медиа
  `?v={rev}` (и `&panel_key=` при пароле): `/shorts/{id}/preview?v={rev}`, `/file?v={rev}`,
  `/thumbnail?v={rev}` — иначе браузер покажет старую версию из кэша после правки.
- **UX правки**: после PATCH — поллить шортс; когда `rev` изменился (и `has_preview`=true) —
  перезагрузить превью с новым `?v=rev`. На время ре-рендера показывать «обновляется…».
- **Одобрение → финал**: `POST /api/shorts/{id}/approve` ставит финальный рендер (1080×1920,
  прожиг субтитров, высокое качество). Это **долго** (~1–3 мин на CPU). `has_final` становится
  `true` ТОЛЬКО когда финал полностью готов (рендер атомарный). Поллить `has_final`; пока false —
  показывать «рендерим финал…», БЕЗ кнопки скачать. Когда `has_final=true` — показать **«Скачать»**
  (`GET /api/shorts/{id}/file?v={rev}`, исходное качество, attachment).
- **Ошибка финала**: если рендер упал — `metadata.render_error` (строка). Показать на approved-без-final
  + кнопку «Повторить» (повторный `POST /approve`).

## 11) Приоритеты провайдеров LLM/STT (упорядоченные списки + фейловер)
Раньше был ОДИН активный провайдер на каждый вид. Теперь — **упорядоченный СПИСОК** (index 0 =
высший приоритет), с авто-фейловером вниз по списку (лимит/кредиты/ошибка → следующий провайдер).

Контракт:
- `GET /api/settings` возвращает `llm_providers: Provider[]` и `stt_providers: Provider[]`
  (по приоритету). `Provider = { id: string, type, base_url?, api_key (маска "****xxxx"),
  model, model_fast?, models?: string[], models_fast?: string[] }`.
- `PUT /api/settings` принимает `llm_providers` / `stt_providers` — **полные упорядоченные списки**.
  - Порядок в массиве = приоритет.
  - Сохранение ключа: у каждого провайдера есть **`id`** — присылай его обратно; если `api_key`
    маскирован/пуст, но `id` совпал — прежний ключ сохранится. Новый провайдер: без `id` (или новый) +
    реальный `api_key`. **Без `id` masked-ключ НЕ сохранится** (создастся новый провайдер с пустым ключом).
- Одиночные `llm_provider`/`stt_provider` = зеркало первого элемента (легаси, можно не трогать).
- `POST /api/providers/test { kind, config }` — тест одного провайдера (как раньше; для «проверить» строки).

UI (Настройки → AI-провайдеры):
- Две секции: **LLM** и **STT**, в каждой — список провайдеров карточками.
- **Перетаскивание для смены приоритета** (drag-and-drop / стрелки вверх-вниз). Подпись: «index 0 —
  основной, ниже — резервные (авто-переключение при лимите/ошибке)».
- Кнопки: добавить провайдер, удалить, «Проверить подключение» на каждую карточку.
- Каждая карточка редактирует type/base_url/api_key/model + списки моделей (model/models — балансир
  моделей внутри провайдера, раздел 1). Пресеты Groq/OpenRouter/Ollama — как раньше.
- На «Сохранить» слать весь упорядоченный список (с `id` у существующих). Ключ вводится только при смене.
- STT: помни, что STT умеют не все (Whisper — Groq/OpenAI/self-hosted; Ollama/OpenRouter — только LLM).

### 11.1) Доп. HTTP-заголовки провайдера (Cloudflare Access и т.п.) — НОВОЕ ПОЛЕ
Некоторые провайдеры (напр. **Alternix Friend API**) требуют кроме `Authorization: Bearer` ещё
пары заголовков Cloudflare Access. Для этого у каждого провайдера появилось поле:
- `extra_headers?: Record<string,string>` — словарь «имя заголовка → значение».
  **Значения — секреты**: в `GET` приходят маскированными (`"****xxxx"`), как `api_key`.
  Сохранение — по тем же правилам: масштабируй логику маски. Если значение маскировано/пусто, но провайдер
  тот же (`id` совпал) — прежнее значение сохранится. Отсутствие ключа `extra_headers` в payload =
  «не трогать»; передача `{}` = очистить.
- UI: на карточке провайдера — сворачиваемый блок **«Доп. заголовки (Cloudflare Access)»** с
  редактируемыми парами ключ/значение (add/remove строки). Значение — поле типа password (маска).
  Удобно дать пресет-кнопку, добавляющую две пустые строки `CF-Access-Client-Id` и `CF-Access-Client-Secret`.
- **Тип провайдера `friend`** (Alternix): OpenAI-совместимый для LLM (`base_url` = `https://friend-api.alt.rent/v1`,
  модель `qwen3:8b`), а для **STT** у него свой эндпоинт (бэкенд знает про это сам по `type:"friend"`;
  `base_url` для STT-карточки — корень `https://friend-api.alt.rent`). Добавь **пресет «Alternix Friend»**
  для обеих секций, который проставляет `type:"friend"`, нужный `base_url` и две пустые строки CF-заголовков.
- Валидацию «STT умеют не все» расширь: `friend` — умеет и LLM, и STT.
